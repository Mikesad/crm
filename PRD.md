# 产品需求文档 (PRD) - 智能企业CRM系统

## 1. 文档基本信息
* **产品名称**：智能企业CRM系统（B2B方向）
* **当前版本**：V1.0.0
* **文档状态**：草稿/评审中
* **面向读者**：后端/前端开发工程师、测试工程师、销售运营总监

---

## 2. 产品概述与核心价值
本项目旨在打造一套覆盖线索、客户、商机、合同、回款全生命周期的轻量级CRM系统。
* **核心痛点**：销售撞单严重、离职客户流失、中高层无法实时掌控销售漏斗及团队业绩。
* **核心价值**：通过严格的 **RBAC功能权限 + 动态数据隔离** 确保客户资产安全；通过**公海池流转机制**提升客户流转效率。

---

## 3. 全局业务流程
整个系统的核心生命周期流转如下：
1. **市场/销售**录入原始【线索】。
2. 跟进后，线索转化为【客户】并自动关联【联系人】。
3. 对客户挖掘出明确需求后，建立【商机】，进入销售漏斗跟进阶段。
4. 商务谈判成功，签署【合同】并制定【回款计划】。
5. 财务录入【回款记录】，订单完成。

---

## 4. 核心功能需求模块

### 4.1 组织架构与动态权限模块

#### 4.1.1 功能描述
管理公司内部组织架构、员工账号，并通过角色控制用户的**功能操作权限**与**数据可见范围**。

#### 4.1.2 权限规则矩阵表
系统必须严格按照下表在**后端拦截器**层面进行数据过滤：

| 序号 | 角色名称 | 默认数据范围 (`data_scope`) | 功能权限边界 |
| :--- | :--- | :--- | :--- |
| 1 | **系统管理员** | 1 - 全部数据权限 | 拥有系统全量配置、账号分配、字典维护权限。 |
| 2 | **销售总监** | 4 - 本部门及以下数据权限 | 查看全盘销售漏斗、修改公海池回收规则、拥有最高分配权。 |
| 3 | **销售主管** | 3 - 本部门数据权限 | 查看并审批下属的合同、商机；可手动划拨团队内的客户。 |
| 4 | **普通销售** | 5 - 仅本人数据权限 | 只能增删改查负责人为自己的数据，**严禁批量导出和物理删除**。 |
| 5 | **财务人员** | 1 - 全部数据（仅限合同/财务模块） | 仅能查看合同与财务回款，无权查看和修改销售的客户/线索跟进细节。 |

---

### 4.2 客户与线索管理模块

#### 4.2.1 线索转客户需求
* **输入触发**：销售对线索跟进后判定为“有意向”，点击【转为客户】按钮。
* **业务逻辑**：
    * 系统在后台利用当前线索数据，自动生成一条【客户表】记录和一条【联系人表】记录。
    * 原线索状态变更为 `3-已转客户`，且在列表中置灰，不可重复转化。
    * 新生成的客户，负责人（`owner_user_id`）继承原线索的负责人。

#### 4.2.2 客户公海池流转机制（自动回收）
* **规则触发**：系统每日凌晨自动执行定时任务（Job）。
* **判断条件**：
    * 若客户不属于公海（`is_public = 0`），且（当前时间 - `last_follow_time` 最终跟进时间） > 15 天。
* **执行动作**：
    * 系统自动将该客户的 `owner_user_id` 置为 `NULL`，并将 `is_public` 修改为 `1`。
    * 在【跟进记录表】中自动插入一条系统日志：*“由于超过15天未跟进，系统自动将该客户回收至公海池。”*

#### 4.2.3 团队协作（客户共享）功能
* **功能描述**：允许销售为了跨部门协作，将自己名下的私海客户共享给其他同事。
* **业务逻辑**：
    * 主销售点击【共享客户】，选择被共享人及权限类型（1-只读 / 2-读写）。
    * 系统往 `crm_customer_share` 表中写入记录。
    * **权限放行**：当被共享人查询客户列表或详情时，后端数据权限组件除了校验 `owner_user_id` 外，还需 `OR` 校验共享表，使其能正常越权查看。

### 4.3 商机与销售漏斗模块

#### 4.3.1 功能描述
通过商机阶段的变化，动态反应公司的销售漏斗健康度。

#### 4.3.2 阶段变更规则
* **商机阶段枚举**：需求分析 → 方案报价 → 商务谈判 → 赢单 / 输单。
* **控制规则**：
    * 每次销售在前端变更商机阶段，必须强制填写一条最新的【跟进记录】，阐述推进详情或输单原因。
    * 当状态变为“赢单”时，系统提示并引导用户直接跳转至【合同新建】页面，自动带出客户和商机基础信息。

---

### 4.4 合同与财务回款模块

#### 4.4.1 合同金额与明细校验
* **业务逻辑**：
    * 签署合同必须选择具体的产品（来自产品库）。
    * 合同总金额（`total_amount`）必须等于商品明细中所有商品（`实际销售价 * 数量`）的总和。
    * **折扣控制**：若实际销售价低于标准售价的 85 折，提交合同时，审批流必须自动升级至“销售总监”进行特殊审批。

#### 4.4.2 回款计划与实际回款关联
* **业务逻辑**：
    * 合同生效后，根据账期生成一条或多条【回款计划】（如首款、尾款）。
    * 财务在收到款项后录入【回款记录】。录入时必须勾选对应的【回款计划】。
    * 当某条计划下的实际回款金额 >= 预计回款金额时，该条计划的状态自动更新为 `2-已回款`。
    * 当合同下的所有回款计划均变为已回款，该合同状态自动自动变更为 `2-已结束`。

### 4.5 跟进中心模块（阶段五新增）

#### 4.5.1 功能描述
为每个销售提供**个人工作台视角**的统一跟进入口，解决当前"导航栏跟进记录是空壳菜单、跟进时间轴仅附在客户详情页"两个未完成的问题。
* **核心价值**：让销售每天打开 CRM 即可看到"今天要跟进什么 / 本周还有哪些没跟进 / 我最近写过什么"，驱动日常拜访节奏。
* **覆盖实体**：**线索、客户、商机、合同**四类业务实体共享同一套跟进记录表（`crm_record`）。
* **复用优先**：沿用 `crm_record` 表与 `/crm/record/*` 接口，本阶段**不新增业务表结构**，仅新增查询接口与前端页面。

#### 4.5.2 跟进中心主视图（路由 `/record/center`）
页面顶部固定展示**待跟进数量徽章**，下方为 3 个并列 Tab：

| Tab | 数据范围 | 排序规则 | 用途 |
| :--- | :--- | :--- | :--- |
| **今日待跟进** | `next_follow_time` ∈ [今日 00:00, 今日 23:59] | 按 `next_follow_time` 升序 | 销售当天必看的行动清单 |
| **本周待跟进（含逾期）** | `next_follow_time` ≤ 本周末 23:59 且 > 今日 23:59；**附加显示逾期项**（`next_follow_time` < 今日 00:00，红色标签） | 逾期优先，再按 `next_follow_time` 升序 | 一周节奏把控 |
| **我的历史** | `create_by = 当前用户` | 按 `create_time` 倒序 | 复盘与跨班次交接 |

> **数据范围说明**：所有 Tab 默认按 `owner_user_id = 当前用户` 过滤（数据权限），与现有 `crm:record:list` 权限码叠加使用。

#### 4.5.3 逾期提醒（前端铃铛 + 红点）
* **入口位置**：主布局右上角铃铛图标，徽章数字 = `next_follow_time < NOW()` 的记录数。
* **触发时机**：登录后首屏加载 + 每 5 分钟轮询 `/crm/record/todo/count` 接口。
* **点击行为**：跳转 `/record/center` 并自动切到"本周待跟进"Tab。
* **零后端成本**：本期**不做**站内信 / 邮件 / 短信通知，仅前端展示。

#### 4.5.4 写作入口（两处统一）
1. **详情页时间轴"+"按钮**：在客户 / 线索 / 商机 / 合同详情页的 `RecordTimeline` 组件右上角，弹出"新增跟进"Dialog。
2. **跟进中心"快速新建"按钮**：位于跟进中心右上角，弹出实体选择器（先选 `related_type`，再选对应业务实体）。

#### 4.5.5 实体关联字段扩展
`crm_record.related_type` 取值升级为枚举：`lead` / `customer` / `business` / `contract`。新写入按新枚举校验；现有记录无 `contract` 类型，无需历史迁移。

#### 4.5.6 跟进迁移规则（线索 → 客户）
当销售将线索转为客户时，原线索下的所有跟进记录按 **模式 A：物理迁移** 处理（**v0.2 决策**）：
* **事务内 UPDATE**：`UPDATE crm_record SET related_type='customer', related_id=? WHERE related_type='lead' AND related_id=?`，与 `LeadService.convertToCustomer` 同事务。
* **同步 INSERT 迁移日志**：新增 `crm_record_migration_log` 表，记录每条跟进从 `lead/原ID` 到 `customer/新ID` 的迁移轨迹，用于审计与反向追溯。
* **同步系统记录**：保留原有的"线索已转化为客户「xxx」"系统记录，按相同迁移规则挂到客户时间轴。
* **核心价值**：客户详情时间轴从首次接触一直延续到当前，一气呵成；聚合查询（跟进数 / 频次）无需去重。
* **不选模式 B（复制追加）的原因**：数据冗余（5 条跟进变 5 条复制 + 1 条汇总）、聚合查询需 `DISTINCT`、原线索被"复活"时两边数据不一致。
* **不选模式 C（独立不动）的原因**：销售查客户完整历史需手动拼 2 个时间轴，UX 割裂。

#### 4.5.7 死线索规则（**v0.2 决策**）
销售可将长期未跟进的线索标记为"已死线索"。原 `crm_lead.status=4` 此前仅作枚举预留，本阶段补完实现：
* **新增字段**：
  * `crm_lead.dead_reason VARCHAR(255) DEFAULT NULL`（死因，可选）
  * `crm_lead.dead_time DATETIME DEFAULT NULL`（标记时间）
* **新增接口**：`POST /api/crm/lead/{id}/markDead`，请求体 `{deadReason?: string}` → 设置 `status=4`、`dead_time=NOW`、`dead_reason=?`。
* **死因可选**：后端不强制必填，但表单显示"建议填写"提示，便于后续报表统计与复盘。
* **死线索写跟进限制**：
  * 后端 `RecordService.append()` 校验 `related_type=lead` 时若 `lead.status=4` → 抛 `BusinessException("死线索不允许写新跟进")`。
  * 前端线索详情时间轴"+"按钮灰化 + tooltip "该线索已死，不可写新跟进"。
* **死线索可查**：原跟进记录全部保留，销售可查看历史复盘"为什么死的"。
* **不删除 `status=4` 枚举的原因**：阶段四已开始使用此状态做筛选项（如列表页下拉），删除会破坏前端兼容性。

#### 4.5.8 软提示：未跟进线索是否允许直接转客户
* **本期决策**（v0.2）：不强制卡点，转客户接口不校验是否已有跟进记录。
* **UI 软提示**：线索详情页"转客户"按钮旁显示橙色警示文案"⚠ 你还未跟进过，建议先写一条跟进"；同时展示"先去跟进"次级按钮，点击直接弹出写跟进 Dialog。
* **理由**：软提示比强制卡点更可持续，销售每次看到警示就会养成"先跟进再转"的习惯，符合 B2B 销售实际节奏。

### 4.6 报表中心模块（阶段五新增）

#### 4.6.1 功能描述
为管理层与销售提供 **4 张业务报表**，纯查询、零写入。
* **数据来源**：各业务表实时聚合，**不另建报表快照表**（避免数据不一致）。
* **核心价值**：让销售总监一眼看清销售漏斗、回款风险；让销售看到自己的跟进频次与赢单率。

#### 4.6.2 报表清单

| # | 报表名称 | 数据维度 | 主图表 | 主要受众 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | **销售漏斗 + 业绩** | 商机阶段金额聚合 + 月度合同/回款金额 | 漏斗图 + 柱状+折线组合图 | 销售总监 / 主管 |
| 2 | **客户分布** | 客户行业 / 来源 / 区域 三维度 | 饼图 + 横向柱状图 | 销售总监 |
| 3 | **跟进与转化率** | 按销售统计跟进频次 / 线索→客户转化率 / 商机赢率 | 柱状图 | 销售总监 / 销售 |
| 4 | **回款 / 财务** | 应收账款账龄 + 超期未回款 + 合同回款进度 | 表格 + 横向柱状图 | 财务 / 销售总监 |

#### 4.6.3 入口与导航
* **侧边栏独立菜单**：新增"报表中心"菜单项，路由 `/report`，内部 4 个 Tab 切换 4 张报表。
* **Dashboard 卡片快入**：在 Dashboard 主页增加 4 张入口卡片，每张卡片直接跳转到对应报表 Tab。

#### 4.6.4 筛选交互（所有报表通用）
* **时间范围筛选**：页面顶部固定时间选择器，支持"今日 / 本周 / 本月 / 本季 / 本年 / 自定义范围"。切换时 4 张报表联动重算。
* **部门 / 销售筛选**：可选"全公司 / 我的部门 / 某销售"下拉。`data_scope = 5` 的销售仅可见"本人"选项。
* **本期不做**：Excel 导出、PDF 打印、邮件订阅。

#### 4.6.5 权限设计（本期简化）
* **本期决策**：报表**对所有登录用户可见**，不叠加 `DataPermissionHandler` 数据权限拦截。
* **原因**：报表是辅助决策工具，初期阶段更看重"全员看到业务全景"的价值；权限分层（销售只能看自己 / 主管看部门）列入后续阶段优化项。
* **安全护栏**：所有报表接口仍走 Sa-Token 鉴权（`crm:report:*` 权限码），未登录用户 401。

#### 4.6.6 技术约束
* **图表库**：复用 `vue-echarts`（阶段二 Dashboard 已有引入）。
* **后端聚合**：报表接口不直接复用业务分页接口，需新增专用聚合查询 SQL（按时间 / 部门切片）。

### 4.7 系统设置模块（阶段六 commit 1 新增）

#### 4.7.1 功能描述
为系统管理员提供**用户、角色、菜单权限、部门**四类基础数据的管理入口。当前阶段一仅把这四张表用作登录态的读数据源，**没有任何写入口**，管理员无法新增员工、调整角色权限或维护组织架构。本模块将这些"运维通道"补齐，作为 CRM 系统可持续运营的基础设施。
* **核心价值**：让 admin 可以在前台完成"开账号 / 调权限 / 改组织"等日常运维，不再依赖 SQL 直连数据库。
* **访问边界**：**系统管理员（admin）+ 销售总监** 2 个角色可见可用；销售主管 / 销售 / 财务均**无权访问**。所有写操作都走 Sa-Token 鉴权（`sys:*:edit` 权限码）+ `@SaCheckRole({"admin","sales_director"})` 双层校验。
    * **理由**：公司体量小，admin 忙不过来时销售总监需能顶岗（开账号 / 调权限 / 改组织），销售总监已有的"4-本部门及以下"数据范围已能查看下属所有客户，顺手管人/管权限合理。
    * **不向销售主管/销售/财务开放**：避免越权（如销售自己改自己的 data_scope）。
* **复用优先**：底层表 `sys_user` / `sys_role` / `sys_menu` / `sys_dept` / `sys_user_role` / `sys_role_menu` 全部已存在；本阶段**仅新增 Java 类（Dept / Role / Menu / User 的 Service+Controller）与前端页面**，不新增业务表。

#### 4.7.2 用户管理（子模块 1）

* **功能列表**：
    * 列表分页（按 username/nickname 关键字、按部门过滤、按状态过滤）。
    * 新建员工：必填 `username / nickname / dept_id / phone / email`，初始密码默认 `123456`（BCrypt 同步落库）。
    * 编辑员工：所有字段可改（除 `username` 外，**`username` 不可改**，作为登录账号稳定不变）。
    * **重置密码**：admin 点击"重置密码"按钮，弹 Dialog 输入新密码（或前端只点确认重置为默认 `123456`），调用 `POST /api/sys/user/{id}/resetPassword`，后端 BCrypt 重加密后 UPDATE。
    * 启停用：`status` 切换 0/1，禁用用户立即踢下线（Sa-Token `StpUtil.logout(userId)`）。
    * **分配角色**：编辑 Dialog 内嵌角色多选（`el-select` multiple 加载全量角色），保存时 `DELETE FROM sys_user_role WHERE user_id=?` + 批量 `INSERT`。
* **安全约束**：
    * admin 不能修改 / 禁用自己（前端按钮灰化 + 后端 `if (targetUserId == StpUtil.getLoginId()) throw 403`）。
    * 至少保留 1 个 `admin` 角色的活跃用户（防误操作把系统锁死，校验 `SELECT count(*) FROM sys_user WHERE id IN (sys_user_role.user_id) AND role_id=1 AND status=1`）。
* **数据权限**：不叠加 `DataPermissionHandler`（admin 天然全量）。

#### 4.7.3 角色管理（子模块 2）

* **功能列表**：
    * 列表分页（按 role_name/role_key 模糊匹配）。
    * 新建 / 编辑：`role_name`（如"销售总监"）、`role_key`（如 `sales_director`，**唯一不可改**）、`data_scope`（1-5 枚举）、`status`。
    * **绑定菜单权限**：角色详情页或编辑 Dialog 内嵌 `el-tree` 复选树（懒加载全量菜单 + 类型 `M/C` 节点可选 + 类型 `F` 节点必选 + 父子联动勾选），保存时全量重绑 `sys_role_menu`。
    * 删除：必须先校验**无用户绑定此角色**（`SELECT count(*) FROM sys_user_role WHERE role_id=?`），否则拒绝。
* **内置角色保护**：`role_key IN ('admin','sales_director','sales_lead','sales','finance')` 这 5 个种子角色**不可删除**（仅可编辑 data_scope 与菜单），防误删导致登录失效。
* **data_scope 说明**（沿用阶段一定义）：
    | 取值 | 含义 | 关联效果 |
    | :--- | :--- | :--- |
    | 1 | 全部 | 不拼数据权限 |
    | 2 | 自定义 | 预留扩展（V1 未启用） |
    | 3 | 本部门 | 拼接 `owner_user_id IN (本部门用户ID列表)` |
    | 4 | 本部门及以下 | 拼接 `owner_user_id IN (本部门及子部门用户ID列表)`，用 `sys_dept.ancestors` 路径解析 |
    | 5 | 仅本人 | 拼接 `owner_user_id = 当前用户` |

#### 4.7.4 菜单权限管理（子模块 3）

* **功能列表**：
    * 树形列表：按 `parent_id` 渲染，缩进 + 图标标识 `M/C/F` 三种类型。
    * 新建 / 编辑：`menu_name` / `parent_id`（父菜单，0 为顶级）/ `order_num` / `path` / `component` / `menu_type` / `perms` / `status`。
    * 删除：必须先校验**无子菜单**（`SELECT count(*) FROM sys_menu WHERE parent_id=?`）且**无角色绑定**（`SELECT count(*) FROM sys_role_menu WHERE menu_id=?`），否则拒绝。
* **菜单类型校验**：
    * `M`（目录）：`perms` 必为空，仅用作层级。
    * `C`（菜单）：`path + component` 必填，关联前端路由。
    * `F`（按钮）：`perms` 必填，关联 `@SaCheckPermission` 注解。
* **核心价值**：让 admin 可以**动态新增功能**（如新加一个"数据看板"）而无需改代码 + 重新部署——`perms` 字段直接被 `StpInterfaceImpl` 加载到 Sa-Token 会话，新接口只需打上对应注解即生效。

#### 4.7.5 部门管理（子模块 4）

* **功能列表**：
    * 树形列表（懒加载或平铺+缩进，参考阶段四公海池"平铺+父级缩进"风格）。
    * 新建 / 编辑：`dept_name` / `parent_id` / `order_num` / `status`。
    * 删除：必须先校验**无子部门**且**无用户归属**（`SELECT count(*) FROM sys_user WHERE dept_id=?`），否则拒绝。
* **`ancestors` 自动维护**：新建/编辑时由 Service 层根据 `parent_id` 拼装 `ancestors`（如 `parent.ancestors + "," + parent.id`），与阶段四"crm_dept"约定一致。
* **核心价值**：组织架构调整（如"华东销售部"改名"华东大客户部"）可在前台一键完成；新增部门立即影响 `data_scope=3/4` 的数据权限拼接。

#### 4.7.6 接口与权限码
* **新增 sys_menu 记录（系统设置 12 条 + 产品分类 2 条 = 14 条）**：
    * **系统设置 12 条**（仅 admin + 销售总监绑）：`sys:system:view` / `sys:user:list` / `sys:user:edit` / `sys:user:reset_pwd` / `sys:user:assign_role` / `sys:role:list` / `sys:role:edit` / `sys:role:assign_menu` / `sys:menu:list` / `sys:menu:edit` / `sys:dept:list` / `sys:dept:edit`。
    * **产品分类 2 条**（**全员 5 角色都绑**——产品分类是公共资源，销售总监/主管/销售/财务/admin 都能看）：`crm:product:category:list` / `crm:product:category:edit`。
    * 详见开发计划"阶段六 commit 1（系统设置 12 条）"与"阶段六 commit 2（产品分类 2 条）"。
* **接口归属**：
    * `GET/POST/PUT/DELETE /api/sys/user` + `POST /api/sys/user/{id}/resetPassword` + `PUT /api/sys/user/{id}/roles`。
    * `GET/POST/PUT/DELETE /api/sys/role` + `PUT /api/sys/role/{id}/menus`。
    * `GET/POST/PUT/DELETE /api/sys/menu`（树形返回）。
    * `GET/POST/PUT/DELETE /api/sys/dept`（树形返回）。
* **既有读接口兼容**：阶段四新增的 `GET /api/sys/user/list`（被共享人下拉用）**保留**，复用 `sys:user:list` 权限码（admin + 销售总监通过 `@SaCheckRole({"admin","sales_director"})` 校验；其他角色通过 `sys:customer:share` 共享模块的鉴权链进入）。

#### 4.7.7 本期不做
* 字典管理（sys_dict）— 业务暂未用到。
* 审计日志（操作流水 / 谁在什么时候改了谁）— 留待阶段七。
* 登录日志 / 失败锁定 / 验证码 — 安全增强。
* 批量导入导出 Excel — admin 体量小，手动更可控。

### 4.8 产品管理模块（阶段六 commit 2 新增/升级）

#### 4.8.1 功能描述
将阶段三已建好的"产品库"从**单层扁平**升级为**SaaS 多版本产品**结构，并补齐产品分类管理。产品为**公共资源**（无 `owner_user_id`），所有角色可读，仅 `crm:product:edit` 可写。
* **核心价值**：让"我们卖的是 SaaS 软件"这件事在 CRM 里**可被建模**——CRM 必须能区分基础版/专业版/旗舰版，并知道每个版本按月/季/年计费。
* **范围划分**：
    * **升级现有**：`crm_product` 增加 `product_line`（套餐线）/ `billing_cycle`（计费周期）两字段；`crm_product_category` 表已存在，补齐 entity/mapper/service/controller。
    * **新增前端**：产品分类页 `/product/category`（独立菜单或抽屉二选一）。
    * **不动**：`/crm/product/*` 既有 5 个 CRUD 接口全部保留，**仅扩展**字段映射。

#### 4.8.2 SaaS 字段升级（`crm_product` 表新增 2 字段）

* **`product_line` VARCHAR(32) DEFAULT '基础版' COMMENT '套餐线'`**
    * 取值枚举：`基础版` / `专业版` / `旗舰版`。
    * 前端下拉单选 + 颜色 chip（基础=灰，专业=蓝，旗舰=金）。
* **`billing_cycle` VARCHAR(16) DEFAULT '年' COMMENT '计费周期'`**
    * 取值枚举：`月` / `季` / `年`。
    * 仅展示，**不参与金额计算**（合同明细仍按 `price × count` 实时反推，阶段三逻辑不变）。
* **数据迁移**：现有种子产品默认 `product_line='基础版'` / `billing_cycle='年'`，新增时由前端表单回填默认值。

#### 4.8.3 产品分类管理（`crm_product_category` 表补齐）

* **既有表结构**：`id / parent_id / category_name`，已建无 Java 类。
* **本期补齐**：
    * `CrmProductCategory` entity + `CrmProductCategoryMapper`。
    * `ProductCategoryService` + `ProductCategoryController`（`GET/POST/PUT/DELETE /api/crm/product/category`）。
    * 前端 `/product/category` 页面（平铺列表 + 父级缩进，参考阶段四 `crm_customer` 列表风格）。
    * **接口语义**：分类为**平铺一级**（不支持子级），前端不做树形，简化心智。
* **产品表单集成**：产品新建/编辑 Dialog 中"产品分类"字段从"按 ID 输入"升级为"下拉选择（来源 `GET /crm/product/category/list`）"。
* **删除约束**：若某分类下有 `crm_product.category_id=此值` 的产品（无论上架/下架/逻辑删除），**禁止删除**（前端禁用 + 后端校验）。

#### 4.8.4 产品列表页升级

* **新增列**：`产品分类`（中文名，从 `categoryId` 联表查 `category_name`）/ `套餐线`（彩色 chip）/ `计费周期`。
* **新增筛选**：按 `categoryId` / `productLine` / `billingCycle` 三个下拉过滤。
* **导出预留**：产品列表右上角"导出"按钮位保留但**disabled + tooltip "暂未开放"**（与阶段五报表导出占位一致）。

#### 4.8.5 交易侧边栏接入

* **位置**：`交易` 分组下顺序为 `合同 / 回款 / 产品`（产品放最后，使用频次最低）。
* **产品管理页面** `/product/list` 路由 + `crm:product:list` 权限码（已存在）— 既有，不动。
* **产品分类页面** `/product/category` 路由 + 新增 `crm:product:category:list` / `crm:product:category:edit` 权限码。
* **详情页**：产品暂不做独立 `/product/:id` 详情页（产品数量少，列表里点"编辑"够用）。

#### 4.8.6 合同侧兼容性

* **合同新建页（`/contract/submit`）**：产品下拉框**新增**"套餐线"列（如"ZenCRM 企业版（旗舰版 / 年）"），让销售签合同时一眼看到版本与计费周期。
* **合同明细存储**：`crm_contract_product` 表**不新增字段**（产品版本/计费周期冗余风险大），明细的 `product_id` 关联回查 `crm_product` 即可。
* **核心价值**：将来报表中心扩展"SaaS 订阅收入分析"（如"2026 年专业版年收入占比"）时，可以直接 `JOIN crm_contract_product + crm_product` 拿到 `product_line` 维度。

#### 4.8.7 本期不做
* 产品多档价格表（不同套餐线不同价）— V1 一个 product 一价够用。
* 座位数 / 用户数上下限 — 合同侧控制。
* 试用期产品（`is_trial`）— 与计费 SaaS 商务模式绑太紧，留待 V2。
* 产品图片 / 富文本描述 — 内部销售用不到。
* 产品 SKU 自动编码生成器 — 人工填 `product_code`，DB 唯一索引兜底。

## 5. 非功能性需求与底层设计约束

### 5.1 数据安全与完整性
* **逻辑删除约束**：为了防止销售因主观原因离职恶意删单，系统内的客户、联系人、商机、合同，在前台点击“删除”时一律执行逻辑删除（更新 `is_deleted = 1`），代码层查询时默认过滤。
* **精密计算约束**：所有涉及金额（如预计金额、合同总金额、回款金额）的后端字段，统一采用 `BigDecimal` 进行加减乘除，严禁出现前端浮点数精度丢失问题。
