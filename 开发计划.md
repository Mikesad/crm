# 智能 CRM 系统后端开发落地计划 (轻量单机版)

作为项目架构师，针对**单机部署、无 Redis 依赖、采用 Sa-Token 鉴权**的敏捷开发场景，我们的设计核心是：**压榨本地内存（如 Sa-Token 的本地缓存组件）的高效性，利用 Spring 内部事件机制替代分布式消息，确保整体架构极简、好维护。**

---

## 一、 核心技术栈选型

针对单机轻量级架构，技术栈全面向“开箱即用、低运维成本”靠拢：

* **核心框架**：Spring Boot 3.x + JDK 17 +vue3 +ElementPlus
* **安全与鉴权**：**Sa-Token（最新轻量版）**
    * *选型理由*：不依赖 Redis 时，Sa-Token 默认使用本地内存存储 Token 映射，其提供的 `StpUtil` 权限拦截和路由拦截器非常轻量，完美契合单机应用。
* **ORM 框架**：MyBatis-Plus
    * *选型理由*：利用其强大的拦截器机制（`DataPermissionInterceptor`）来实现 PRD 中要求的数据可见范围隔离。
* **数据库**：MySQL 8.0（所有关联均采用**逻辑外键**，利用物理索引 `KEY` 优化查询）。
* **本地缓存**：Sa-Token 默认内存存储（或按需引入 Caffeine），用于存放临时登录态及权限列表。
* **定时任务**：Spring 自带的 **`@Scheduled`**（单机部署无需分布式任务调度引擎，自带定时器即可满足凌晨的数据处理）。

---

## 二、 开发计划与业务落地拆解

根据“**核心且简单优先 > 核心且难优先 > 边缘且简单 > 边缘且难**”的架构原则，开发周期划分为四个阶段（已剔除纯常规 CRUD 逻辑）。

### 阶段一：基石与骨架构建（最高优先级）
> **策略**：虽然 Sa-Token 接入很简单，但**动态数据隔离**是 CRM 的灵魂，必须在第一行具体业务代码编写前在底层立住，否则后期重构成本极大。

1. **Sa-Token 基础权限拦截（简单 - 核心）**
   * **业务实现**：配置 Sa-Token 全局过滤器/拦截器，拦截未登录请求。
   * **逻辑**：实现 `StpInterface` 接口。当用户登录成功后，Sa-Token 会自动调用该接口。从数据库（`sys_user_role` & `sys_role_menu`）中捞出当前用户的角色和权限标识（如 `crm:customer:delete`）并缓存在本地内存中，供 `@SaCheckPermission` 注解进行鉴权。

2. **MyBatis-Plus 动态数据权限拦截（困难 - 核心）**
   * **业务实现**：编写自定义的 `DataPermissionHandler`。
   * **逻辑**：通过 Sa-Token 的 `StpUtil.getLoginId()` 获取当前登录销售的 ID。解析其角色的 `data_scope`。
     * 若为“仅本人（5）”，动态在客户、商机等 SQL 末尾强行拼接 `WHERE owner_user_id = 个人ID`。
     * 若为“本部门（3）”，则拼接 `WHERE owner_user_id IN (本部门用户ID列表)`。

---

### 阶段二：客户资产流转跟进（高优先级）
> **策略**：打通核心主干业务流，让销售人员能够录入、跟进和转化客户。

1. **线索一键转客户（简单 - 核心）**
   * **业务实现**：使用 Spring 的声明式事务（`@Transactional`）。
   * **逻辑**：接收线索 ID，校验其状态是否为未转化。从 `crm_lead` 复制数据，同步向 `crm_customer` 和 `crm_contact` 表双写数据。成功后将原线索状态更新为 `3-已转客户`。
   * *单机优势*：单机本地事务，无需考虑分布式事务，`@Transactional` 即可保证 100% 强一致性。

2. **跟进时间轴聚合（简单 - 核心）**
   * **业务实现**：拦截客户、商机状态变更的 API（或在 Service 层埋点），销售每次修改阶段时，强制要求传入跟进内容，并异步写入 `crm_record` 表，形成以 `related_id` 为轴心的时间线。

3. **商机漏斗推进（简单 - 核心）**
   * **业务实现**：控制商机阶段（需求分析 → 方案报价 → 商务谈判 → 赢单/输单）的单向流转。当状态变更为“赢单”时，后端返回特定标识，引导前端直接跳转至“新建合同”页面。

---

### 阶段三：资金与合规管控（中等优先级）
> **策略**：涉及金额与财务，难点在于数值的精确计算与多表状态的联动。

1. **合同与产品金额防篡改（中等难度 - 核心）**
   * **业务实现**：后端通过前端传来的 `product_id` 集合，去 `crm_product` 表实时查出标准单价，配合折扣率（`discount`）在后端使用 `BigDecimal` 重新计算总价，校验是否等于前端传来的 `total_amount`。
   * **合规逻辑**：若计算出的实际折扣低于 85 折，提交合同时自动将状态置为 `0-审批中`，并向其主管发送审批待办。

2. **回款计划与实收联动核销（中等难度 - 核心）**
   * **业务实现**：财务录入 `crm_receivable`（实际回款）时，关联 `crm_receivable_plan`（计划 ID）。
   * **联动逻辑**：单机环境下，直接利用 **Spring Event（ApplicationEventPublisher）** 发送一条“回款成功事件”。合同服务监听到事件后，累加该计划下的总实收。若实收款 $\ge$ 计划款，更新计划状态为已回款；同时检查当前合同下的所有计划是否均已完成，若完成则自动将 `crm_contract` 状态变更为 `2-已结束`。

---

### 阶段四：高级业务特性（低优先级，延后迭代）
> **策略**：属于旁路优化和提升协作效率的功能，可放在系统整体闭环跑通后再做。

1. **公海池自动回收任务（中等难度 - 非核心）**
   * **业务实现**：在单机架构中，直接在启动类开启 `@EnableScheduling`，并编写一个定时方法 `@Scheduled(cron = "0 0 2 * * ?")`（每天凌晨2点执行）。
   * **逻辑**：扫描 `crm_customer` 表中 `is_public = 0` 且 `last_follow_time` 距离当前时间超过 15 天的记录。将其 `owner_user_id` 置为 `NULL`，`is_public` 置为 `1`。
2. **跨部门客户共享放权（困难 - 非核心）**
   * **业务实现**：针对 `crm_customer_share` 表进行权限穿透。
   * **难点**：需要回头修改阶段一写好的“MyBatis-Plus 数据权限拦截器”。当普通销售查询客户列表时，拦截器拼装的 SQL 需从 `WHERE owner_user_id = 当前用户` 升级为：
     `WHERE owner_user_id = 当前用户 OR id IN (SELECT customer_id FROM crm_customer_share WHERE user_id = 当前用户)`。
     需要精细调试动态 SQL 解析器，防止产生 SQL 语法错误。

---

### 阶段五：跟进与报表（拆为两个独立 commit）

> **策略**：阶段五拆为两个可独立交付的 commit，**commit 1：跟进中心**（打通个人工作台）→ **commit 2：报表中心**（管理层决策支持）。两个 commit 之间可被任意打断、单独验收。
> **为什么不做阶段五再拆两个阶段**：阶段五两个模块都围绕"已沉淀数据二次利用"，工程量约 3-4 天，单独再分阶段会拖慢节奏。

##### 4.5) 线索-客户跟进迁移（v0.2 决策）
* **迁移规则**：`LeadService.convertToCustomer` 在 `@Transactional` 事务内追加两步：
  1. `UPDATE crm_record SET related_type='customer', related_id=#{newCustomerId} WHERE related_type='lead' AND related_id=#{leadId}`
  2. 保留原有"线索已转化为客户「xxx」"系统记录的 INSERT，但相关字段同步迁移到 customer 视角
* **新增迁移日志表**（**新增表结构，须同步两份 SQL**）：
  ```sql
  CREATE TABLE crm_record_migration_log (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    record_id     BIGINT       NOT NULL,
    from_type     VARCHAR(16)  NOT NULL,
    from_id       BIGINT       NOT NULL,
    to_type       VARCHAR(16)  NOT NULL,
    to_id         BIGINT       NOT NULL,
    migrate_time  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    operator_id   BIGINT       NOT NULL,
    INDEX idx_record (record_id),
    INDEX idx_to (to_type, to_id)
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='跟进记录迁移日志';
  ```
* **新增权限码**：`crm:lead:mark_dead`、`crm:lead:read`（线索详情页基础权限，沿用现有 `crm:lead:*` 即可）

##### 4.6) 死线索实现（v0.2 决策）
* **crm_lead 表新增字段**（**走 phase5_add_col_if_missing 存储过程**，同步两份 SQL）：
  ```sql
  ALTER TABLE crm_lead
    ADD COLUMN dead_reason VARCHAR(255) DEFAULT NULL COMMENT '死因,可空',
    ADD COLUMN dead_time   DATETIME     DEFAULT NULL COMMENT '标记时间';
  ```
* **新增接口** `POST /api/crm/lead/{id}/markDead` body `{deadReason?: string}`：
  * 校验：当前用户是 `lead.owner_user_id`（防越权）；`lead.status` 必须 ∈ {1, 2}（未跟进/跟进中可标死，已转客户/已死不可重复标）。
  * 动作：UPDATE `crm_lead SET status=4, dead_time=NOW(), dead_reason=? WHERE id=?`。
  * 同步：写一笔系统跟进 `crm_record(relatedType='lead', content='线索标记为死线索:「xxx」', followType='system')`。
* **后端写跟进限制**：`RecordService.append()` 在白名单校验后追加：若 `relatedType='lead'` 且 `lead.status=4` → 抛 `BusinessException("死线索不允许写新跟进")`。
* **前端**：
  * `views/lead/detail.vue`（**本 commit 一并新建**，是 commit 1 的"详情页最小可读骨架"必建项之一）头部加"标记为死线索"按钮（owner 可见），点击弹 Dialog：死因下拉（不感兴趣 / 联系方式错误 / 已被竞品签下 / 预算不足 / 其他）+ 备注 textarea + 确认按钮。
  * 线索详情页 `status=4` 时：顶部红色 `el-alert` "该线索已死，不可写新跟进"；时间轴"+"按钮灰化 + tooltip。
  * 死线索时间轴可查看、可展开"系统记录"那条死因。
* **死线索列表筛选**：`lead/list.vue` 已有"已死线索"下拉筛选项，确认后端 `LeadQuery` DTO 已支持 `status=4`（状态机已就绪，仅新增字段）。

#### Commit 2：报表中心

##### 6) 业务目标
* 4 张业务报表：销售漏斗+业绩 / 客户分布 / 跟进与转化率 / 回款+财务。
* **双入口**：侧边栏独立菜单 + Dashboard 卡片快入。
* 顶部时间范围 + 部门/销售筛选，所有报表联动重算。
* **本期所有人可见**，不叠加 `DataPermissionHandler` 数据权限拦截。

##### 7) 后端实现
* **新增 5 个聚合接口**：
  * `GET /crm/report/funnel?startDate=&endDate=&deptId=&userId=` → 销售漏斗（5 个阶段金额 + 商机数）。
  * `GET /crm/report/performance?startDate=&endDate=` → 月度业绩（合同金额、回款金额、按月分桶）。
  * `GET /crm/report/customer-distribution?dimension=industry|source|region` → 客户分布（按维度返回 Top 10 + 其他）。
  * `GET /crm/report/followup-conversion?startDate=&endDate=&deptId=&userId=` → 跟进与转化率（跟进频次 / 线索转客户率 / 商机赢率）。
  * `GET /crm/report/receivable-aging?asOfDate=` → 应收账款账龄（30 / 60 / 90 / 90+ 天分桶 + 合同回款进度）。
* **聚合 SQL 策略**：
  * **单 SQL + GROUP BY 维度**，避免 N+1 多次往返。
  * 时间范围统一在 Service 层转 `LocalDate` 区间，避免日期函数不能走索引。
  * 数据量大时 `LIMIT 500` 兜底（不让报表拖垮 DB）。
  * 索引补充：`crm_contract(sign_date)`、`crm_receivable(actual_time)`、`crm_business(stage, expected_deal_date)`。
* **新增 VO**：`FunnelVO` / `PerformanceVO` / `CustomerDistributionVO` / `FollowupConversionVO` / `ReceivableAgingVO`。
* **权限码**：`crm:report:funnel` / `crm:report:performance` / `crm:report:customer-distribution` / `crm:report:followup-conversion` / `crm:report:receivable-aging`。

##### 8) 前端实现
* **路由**：`/report` → `views/report/index.vue`，4 个 Tab（与后端 4 个报表一一对应）。
* **新增子组件**：
  * `views/report/components/FunnelChart.vue`（vue-echarts 漏斗图）。
  * `views/report/components/PerformanceChart.vue`（柱状+折线组合图）。
  * `views/report/components/CustomerDistributionChart.vue`（饼图 + 横向柱状图）。
  * `views/report/components/FollowupConversionChart.vue`（柱状图）。
  * `views/report/components/ReceivableAgingTable.vue`（Element Plus Table + 横向进度条）。
* **复用筛选条**：抽象 `components/ReportFilterBar.vue`（时间范围 + 部门/销售下拉树 + 应用按钮）。
* **Pinia 缓存**：`useReportStore` 按 `(reportKey + filterHash)` 缓存 5 分钟，避免切换 Tab 重复请求。
* **Dashboard 改造**：`views/dashboard/index.vue` 增加 4 张报表入口卡片，点击跳转 `/report?tab=funnel` 之类。

##### 9) SQL / 菜单变更
* `sys_menu` 新增 1 条：`报表中心` → 路由 `/report`，权限码 `crm:report:*`。
* `sys_role_menu` 全部角色绑定。
* **同步更新两份 SQL**：`sql/crm_full.sql` + `sql/migrations/phase5-report.sql`（同样 `DELETE + INSERT WHERE perms IN ('crm:report:funnel', ...)` 幂等模式）。
* **新建索引**：上述 3 个索引列在两份 SQL 中同步 `CREATE INDEX IF NOT EXISTS`（按阶段迁移脚本约定走 `phase5_add_index_if_missing` 存储过程）。

##### 10) 本 Commit 不做
* 数据权限拦截（所有人可见，权限分层列入阶段六）。
* Excel / CSV 导出。
* 报表订阅、邮件推送、定时报表。
* 报表快照表（聚合查询实时跑，避免数据不一致）。

##### 11) 验收标准
* 4 个报表接口 P95 响应 < 1s（参考值，超出加索引）。
* 报表页面切换 Tab 无白屏（store 缓存命中）。
* 时间范围 / 部门筛选实时联动，4 张图同步重算。
* 角色权限：未登录用户访问 `/report` → 401。

##### 12) 风险与对策
| 风险 | 等级 | 对策 |
| :--- | :--- | :--- |
| 聚合 SQL 拖垮 DB | 中 | 索引补充 + LIMIT 500 + 前端 5min 缓存 |
| 部门/销售树形数据来源 | 低 | 复用阶段一 `sys_dept` / `sys_user` 表，前端缓存组织树 |
| ECharts 主题统一 | 低 | 复用 Dashboard 阶段二已沉淀的 echarts theme，禁止组件内硬编码色值 |
| 大数据量报表导出 | 低 | 本期不做导出，留出"导出"按钮位但 disabled 加 tooltip 提示"暂未开放" |
